# LangGraph Persistence and HITL Operations

## Purpose
Operationalize LangGraph persistence and interrupt/resume rules for this project.

## Thread and Checkpoint Policy
- For graph runs compiled with a checkpointer, every invocation must include:
  - `configurable.thread_id` (required)
  - optional `configurable.checkpoint_id` for replay/debug
- No anonymous runs in production paths.
- Super-step checkpoint boundaries are considered authoritative recovery points.

## Required Runtime Capabilities
- `get_state(config)` for latest checkpoint snapshot.
- `get_state_history(config)` for timeline/replay/debug.
- Replay from prior checkpoints for incident analysis.
- Controlled `update_state` path for operator interventions.
- Explicit delete-thread operation support for controlled cleanup workflows.

## Checkpointer Matrix
- Dev/unit tests: `InMemorySaver`
- Local durable integration: `SqliteSaver` / `AsyncSqliteSaver`
- Production: `PostgresSaver` / `AsyncPostgresSaver`
- Optional cloud/provider-specific implementations as infra requires.

## Store Policy (Cross-Thread Memory)
- Use checkpointer for per-thread state.
- Use Store for cross-thread memory (user/run scoped namespaces).
- Namespace convention:
  - `("memories", user_id)` for user-level memory
  - `("run_artifacts", run_id)` for run-scoped durable notes
- Production store must be persistent; in-memory store only for test/dev.

## HITL Interrupt Contract
- Pauses must be generated by `interrupt(...)` in node/tool code.
- Resume only through `Command(resume=...)` with same `thread_id`.
- Supported decision actions:
  - `approve`
  - `edit`
  - `reject`
  - `feedback`
- Resume payload schema validation is mandatory before graph invocation.
- `Command(update=...)` must not be used as input for conversational continuation; only `Command(resume=...)` is valid input for resumes.
- Multiple simultaneous interrupts must be resumable via interrupt-id -> response map.

## Interrupt Safety Rules
- Do not wrap interrupts in broad `try/except`.
- Keep interrupt call order deterministic within a node.
- Use JSON-serializable payloads only.
- Side effects before interrupt must be idempotent or moved to later nodes.
- Nodes that may re-enter after interrupt must not perform non-idempotent pre-interrupt writes.

## Update-State Governance
- `update_state` is restricted to:
  - debugging/time-travel experiments
  - explicit operator overrides
- All updates must record:
  - actor id
  - reason
  - previous checkpoint id
  - resulting checkpoint id
  - `as_node` if used

## Dashboard Requirements (Persistence-Aware)
- Display active `thread_id`, latest `checkpoint_id`, `checkpoint_ns`.
- Show pending interrupts and their ids.
- One-click load of checkpoint history for current thread.
- Replay controls:
  - replay from checkpoint
  - fork run from checkpoint
  - compare branch outcomes
- Streaming inspector should support LangGraph v2 event shape (`type`, `ns`, `data`) and interrupt extraction.

## Audit Events (Minimum)
- `thread_started`
- `checkpoint_written`
- `interrupt_issued`
- `interrupt_resumed`
- `state_updated`
- `replay_started`
- `replay_completed`
- `checkpoint_forked`

## Security and Encryption
- Enable encrypted checkpoint serialization in sensitive environments.
- Never store secrets in interrupt payloads.
- Redact sensitive state fields before exporting traces externally.

## CI and Validation
- Contract tests for interrupt/resume payload compatibility.
- Integration tests for same-thread resume requirement.
- Replay tests asserting nodes before checkpoint are skipped.
- Tests for pending-write behavior and failure recovery.
- Tests for multiple parallel interrupt resume mapping.
- Tests that broad exception handlers do not swallow interrupt control flow.
